<task type="auto">
  <name>Task 1: Create GitHub Actions CI workflow</name>
  <files>.github/workflows/ci.yml</files>
  <action>
Create `.github/workflows/ci.yml` with the following configuration:

**Name:** `CI`

**Triggers:**
- `push` to `main` branch
- `pull_request` to `main` branch

**Concurrency:** Group by `ci-${{ github.ref }}`, cancel in-progress on same ref.

**Job: `ci`**
- `runs-on: ubuntu-latest`
- **Steps:**
  1. `actions/checkout@v4`
  2. `actions/setup-node@v4` with `node-version-file: '.nvmrc'`
  3. `oven-sh/setup-bun@v2` with `bun-version-file: '.bun-version'`
  4. `bun install --frozen-lockfile`
  5. **Lint:** `bun run lint`
  6. **Typecheck:** `bun run tsc --noEmit`
  7. **Test:** `bun run test:run`
  8. **Build:** `bun run build` (with `env: { NEXT_PUBLIC_SUPABASE_URL: "https://placeholder.supabase.co", NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY: "placeholder", NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY: "placeholder", NEXT_PUBLIC_SITE_URL: "http://localhost:3000" }` — the build needs these env vars to compile but they are not used in CI)
  9. **Audit (non-blocking):** `bun audit || true` with `continue-on-error: true` (bun audit may not support all npm audit features yet)

**Important:** The build step needs `NEXT_PUBLIC_*` env vars set to dummy values because Next.js inlines them at build time. Without them, the build fails. These are NOT secrets — they are public client-side keys. Real values are only needed in production.

Do NOT add `env` at the job level for Sentry variables — the `next.config.ts` already has `silent: !process.env.CI` which suppresses Sentry source map upload errors when `SENTRY_AUTH_TOKEN` is not set.
  </action>

  </verify>

<task type="auto">
  <name>Task 2: Create Dependabot configuration</name>
  <files>.github/dependabot.yml</files>
  <action>
Create `.github/dependabot.yml` with:

```yaml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    open-pull-requests-limit: 5
    groups:
      production:
        dependency-type: "production"
        update-types:
          - "minor"
          - "patch"
      development:
        dependency-type: "development"
        update-types:
          - "minor"
          - "patch"
    labels:
      - "dependencies"
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    open-pull-requests-limit: 3
    labels:
      - "dependencies"
      - "github-actions"
```

This configures:
- Weekly npm dependency updates grouped by production/dev, limited to minor+patch (major updates are too risky for auto-PR)
- Weekly GitHub Actions updates (keeps CI action versions current)
- 5 open PR limit for npm, 3 for GH Actions
  </action>

  </verify>
  <done>Dependabot configured for weekly npm and GitHub Actions updates with grouped PRs</done>
