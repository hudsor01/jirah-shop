<objective>
Eliminate the N+1 order count pattern on the admin customers page.

Purpose: The current code fetches ALL order rows matching a page's customer IDs (potentially thousands of rows), then reduces to counts in JS. While it's technically a single query (not N+1 per-customer), it downloads all matching order rows just to count them. The fix keeps the single query approach but uses Supabase's count-by-group capabilities more efficiently. Since Supabase JS client doesn't natively support `GROUP BY` with `.select()`, the most practical fix is to keep the current pattern (single `.in()` query + JS reduce) but ensure it only selects the `user_id` column (which it already does) -- making it lightweight. The real N+1 risk was if it queried per customer; the current code is already grouped into one query.

After review: The current code already uses a single query with `.select("user_id").in("user_id", customerIds)` and reduces in JS. This is NOT an N+1 -- it's a single query selecting only the user_id column. The fix should be minimal: add a `.limit()` guard or switch to an RPC if the row count could be very large. For this store's scale, the current approach is acceptable. The plan verifies this and documents it.

Output: Verified and potentially optimized customers page query. Minimal code changes.
</objective>
