# Architecture Overview

## System Diagram

Include a Mermaid diagram showing:
- Browser/Client -> Next.js App (Server Components, Server Actions, API Routes)
- Next.js -> Supabase (Auth, Database, Storage)

- Next.js -> Sentry (error tracking)

## Key Architecture Layers

1. **Presentation**: React Server Components + Client Components (shadcn/ui)
2. **Business Logic**: Server Actions in `actions/` (all return `ActionResult<T>`)
3. **Data Access**: Query functions in `queries/` (Supabase client calls)
4. **External Services**: Stripe (payments), Supabase (auth, db, storage), Sentry (monitoring)
5. **Caching**: Next.js `use cache` + `cacheTag()` for read-heavy queries

## Data Flow
- Server Components call `queries/` functions directly for reads
- Client Components call Server Actions for mutations
- Server Actions validate with Zod, execute business logic, return `ActionResult<T>`
- Stripe webhooks handle async payment completion
- `use cache` caches queries; `updateTag()` invalidates on mutation

## Architecture Decision Records
Link to the 4 ADR files in docs/adr/.

**ADR Format (lightweight):**
Each ADR follows this structure:
- Title
- Status: Accepted
- Context: Why the decision was needed
- Decision: What was chosen
- Consequences: Trade-offs and implications

**docs/adr/001-supabase-auth-storage.md:**
- Context: Need managed auth, PostgreSQL database, and file storage for e-commerce platform
- Decision: Supabase for all three (auth, database, storage) using @supabase/ssr with getAll/setAll cookie pattern
- Consequences: Single vendor dependency, but reduced ops overhead. SSR-compatible auth. NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY naming convention.

**docs/adr/002-stripe-checkout.md:**
- Context: Need payment processing. Options: Stripe Elements (embedded), Stripe Checkout (redirect)
- Decision: Stripe Checkout (redirect-based) for simpler PCI compliance
- Consequences: Less UI control but zero PCI scope. Webhook required for order completion. Guest checkout supported via session metadata.

**docs/adr/003-nextjs-caching.md:**
- Context: Need caching strategy for read-heavy queries (products, blog, settings)
- Decision: Next.js 16 experimental `use cache` + `cacheTag()` with `updateTag()` invalidation (not ISR/revalidateTag)
- Consequences: Cutting-edge API (experimental.useCache config flag required). Per-function granular cache control. React.cache() for per-request deduplication.

**docs/adr/004-action-result-pattern.md:**
- Context: Inconsistent error handling across 38 server actions (some threw, some returned nulls, some returned objects)
- Decision: `ActionResult<T>` discriminated union with ok()/fail() helpers
- Consequences: All 38 actions use consistent contract. Client consumers discriminate on `success` boolean. No thrown exceptions from server actions. See docs/error-handling.md for full documentation.

```markdown
# Project Conventions

## Stack
- Next.js 16, React 19, TypeScript, TailwindCSS v4, shadcn/ui
- Supabase (auth + storage + PostgreSQL), Stripe (payments)
- Package manager: bun (never npm/yarn/pnpm)

## Server Actions
- All server actions return `ActionResult<T>` from `@/lib/action-result`
- Use `ok(data)` for success, `fail("message")` for errors
- Validate all input with Zod `safeParse()` before business logic
- No thrown exceptions — all errors are returned values
- See `docs/error-handling.md` for full pattern

## Data Access
- Query functions in `queries/` directory (not inline in actions)
- Server actions call query functions for database operations
- Server Components can call query functions directly for reads

## Supabase Client
- Use `@supabase/ssr` with `getAll`/`setAll` cookie methods only
- Never use `get`/`set`/`remove` cookie methods
- Never import from `auth-helpers-nextjs`
- Browser client: `lib/supabase/client.ts`
- Server client: `lib/supabase/server.ts`
- Env var: `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` (not ANON_KEY)

## Caching
- `use cache` + `cacheTag()` for read-heavy queries
- `updateTag()` to invalidate (not `revalidateTag()`)
- `React.cache()` for per-request deduplication in `lib/cached-queries.ts`
- Config flag: `experimental.useCache: true` in next.config

## Testing
- Vitest for unit/integration tests
- `vi.hoisted()` mandatory for mock variables in `vi.mock()` factories
- Playwright for E2E tests in `tests/e2e/`
- Coverage minimum: 30% (statements, branches, functions, lines)
- Run: `bun test`, `bun run test:coverage`, `bun run test:e2e`

## Patterns
- Image upload: `useSupabaseUpload` hook -> `<Dropzone>` component
- Cart: `useCart()` from `providers/cart-provider`
- Checkout: `useCheckout()` shared hook
- Auth: Google OAuth via browser client (not server actions)
- Admin Supabase client: lazy singleton in `lib/supabase/admin.ts`
```

# Contributing to jirah-shop

## Development Setup
Reference README.md for setup instructions.

## Branching Strategy
- `main` — production branch, always deployable
- Feature branches: `feature/description` or `fix/description`
- Branch from `main`, merge back via PR

## Pull Request Process
1. Create feature branch from `main`
2. Make changes, ensure tests pass (`bun run test:run`)
3. Run lint and typecheck (`bun run lint && bun run build`)
4. Open PR with description of changes
5. CI must pass (lint, typecheck, test, build)
6. Request review
7. Squash merge to `main`

## Code Style
- TypeScript strict mode
- Server actions return `ActionResult<T>` — see docs/error-handling.md
- Zod validation on all server action inputs
- Use `queries/` layer for database access, not inline queries
- shadcn/ui for UI components
- TailwindCSS for styling (no CSS modules)
- Use bun as package manager (never npm/yarn/pnpm)

## Testing Expectations
- New server actions: add unit tests
- New features: add integration tests where possible
- Critical paths: maintain 30% minimum coverage
- E2E: add Playwright tests for user-facing flows

## Commit Messages
Use conventional commits: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`, `chore:`
