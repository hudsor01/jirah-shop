# Error Handling: ActionResult&lt;T&gt;

## Overview
All server actions in jirah-shop return `ActionResult<T>` — a discriminated union that makes error handling explicit and type-safe. There are no thrown exceptions from server actions.

## The Type
```typescript
// lib/action-result.ts
type ActionResult<T = void> =
  | { success: true; data: T }
  | { success: false; error: string };
```

## Helper Functions
```typescript
import { ok, fail } from "@/lib/action-result";

// Success
return ok(product);     // { success: true, data: product }
return ok(undefined);   // { success: true, data: undefined } (for void actions)

// Failure
return fail("Not found");  // { success: false, error: "Not found" }
```

## Server Action Pattern
Show a typical server action pattern with:
- Zod validation at the top
- Business logic returning ok() or fail()
- No try/catch needed for expected errors

```typescript
"use server";
import { ok, fail, type ActionResult } from "@/lib/action-result";

export async function createThing(data: FormData): Promise<ActionResult<Thing>> {
  const parsed = thingSchema.safeParse(Object.fromEntries(data));
  if (!parsed.success) return fail("Invalid input");

  const result = await db.things.insert(parsed.data);
  if (!result) return fail("Failed to create thing");

  return ok(result);
}
```

## Client Consumption Pattern
Show how client components consume ActionResult:

```typescript
"use client";
import { createThing } from "@/actions/things";

function CreateForm() {
  async function handleSubmit(formData: FormData) {
    const result = await createThing(formData);
    if (result.success) {
      toast.success("Created!");
      // result.data is typed as Thing
    } else {
      toast.error(result.error);
      // result.error is typed as string
    }
  }
}
```

## Key Principles
- **No thrown exceptions**: All errors are returned as values
- **Discriminated union**: TypeScript narrows the type based on `success` boolean
- **Generic data type**: `ActionResult<Product>`, `ActionResult<void>`, `ActionResult<Order[]>`
- **Consistent error strings**: Human-readable messages safe to display in toasts
- **No sensitive data in errors**: Error messages never contain stock counts, coupon codes, or internal state

## Migration from try/catch
Brief note: The project migrated from ad-hoc error handling to ActionResult in Phase 6. All 38 server actions now use this pattern.
