2. **After each creation step, store the ID:**
   - After `stripe.products.create()`: set `stripeProductId = stripeProduct.id`
   - After `supabase.from("products").insert()`: set `dbProductId = product.id`
   - After Stripe price creation: no separate tracking needed (prices are tied to the Stripe product -- archiving the product handles them)

3. **For the DB product insert failure**, explicitly throw so the catch block runs:
   - Change from `if (error) { return { success: false, error: error.message }; }` to `if (error) throw new Error(error.message);`

4. **For the variant insert**, make the existing variant error non-silent. Currently variant errors are only logged but the function returns success. Change to:
   - If `variantError` occurs: throw `new Error(\`Variant creation failed: ${variantError.message}\`)`
   - This ensures cleanup runs when variants fail

5. **In the catch block**, add reverse-order cleanup BEFORE the return:

```typescript
} catch (err) {
  // Cleanup in reverse order of creation
  if (dbProductId) {
    const { error: cleanupDbError } = await supabase
      .from("products")
      .delete()
      .eq("id", dbProductId);
    logger.error("createProduct rollback: deleted Supabase product", {
      dbProductId,
      cleanupSuccess: !cleanupDbError,
    });
  }
  if (stripeProductId) {
    try {
      await stripe.products.update(stripeProductId, { active: false });
      logger.error("createProduct rollback: archived Stripe product", {
        stripeProductId,
      });
    } catch (stripeCleanupErr) {
      logger.error("createProduct rollback: failed to archive Stripe product", {
        stripeProductId,
        error: stripeCleanupErr instanceof Error ? stripeCleanupErr.message : "Unknown",
      });
    }
  }
  const message = err instanceof Error ? err.message : "Unknown error";
  return { success: false, error: message };
}
```

Key rules:
- Use `stripe.products.update(id, { active: false })` to archive, NOT `stripe.products.del()` (deletion fails when prices exist)
- Wrap Stripe cleanup in its own try/catch so a cleanup failure doesn't mask the original error
- Log ALL cleanup actions via `logger.error()` so they appear in Sentry for audit
- The existing `requireAdmin()` call and `revalidatePath()` calls stay unchanged
- Do NOT modify `updateProduct` or `deleteProduct` functions -- only `createProduct`
- Cleanup is sequential (not parallel) -- simpler and order is guaranteed
